# Backend Roadmap Hero — учебный разбор

Небольшой, но «взрослый» по устройству backend: авторизация по email/паролю
и сохранение прогресса обучения. Стек: **Express 5 · TypeScript · SQLite
(встроенный `node:sqlite`, Node 22+) · zod · JWT**.

## Запуск

```bash
cd server
npm install
npm run dev        # http://localhost:3001, перезапуск при изменениях (tsx watch)
```

Фронтенд запускается отдельно (`npm run dev` в корне) и проксирует `/api`
на этот сервер (см. `vite.config.ts`). `.env` опционален — см. `.env.example`.

Production: `npm run build && NODE_ENV=production JWT_SECRET=... npm start` —
один процесс раздаёт и API, и собранный фронтенд из `../dist`.

## Карта файлов

```
src/
  index.ts                  точка входа: запуск сервера, больше ничего
  app.ts                    сборка приложения: middleware → роуты → error handler
  config.ts                 конфигурация из переменных окружения, в одном месте
  db.ts                     подключение SQLite + создание схемы
  schemas.ts                zod-схемы: валидация всего, что приходит по сети
  lib/
    password.ts             хеширование паролей (scrypt + соль)
    jwt.ts                  подпись и проверка JWT
    http-error.ts           ошибка со статусом для error handler'а
  middleware/
    auth.ts                 requireAuth: пускает только с валидным токеном
    error-handler.ts        все ошибки → единый JSON-формат { error }
    rate-limit.ts           защита /api/auth от перебора паролей
  repositories/
    users.repo.ts           весь SQL по пользователям
    progress.repo.ts        весь SQL по прогрессу
  routes/
    auth.routes.ts          register / login / logout / me
    progress.routes.ts      GET и PUT прогресса
```

Путь запроса: **express.json → cookie-parser → (rate-limit) → роут →
(requireAuth) → zod-валидация → репозиторий → ответ**, а любая ошибка
на любом шаге проваливается в error-handler.

## API

| Метод | Путь | Тело | Ответ |
| --- | --- | --- | --- |
| POST | `/api/auth/register` | `{ email, password }` | `201 { user }` + cookie |
| POST | `/api/auth/login` | `{ email, password }` | `{ user }` + cookie |
| POST | `/api/auth/logout` | — | `{ ok }`, стирает cookie |
| GET | `/api/auth/me` | — | `{ user }` или `401` |
| GET | `/api/progress` 🔒 | — | `{ lessons, exercises, tasks, updatedAt }` |
| PUT | `/api/progress` 🔒 | `{ lessons, exercises, tasks? }` | `{ ok }` |

Поле `tasks` (чек-лист заданий с лендинга) в PUT опционально: старый клиент
без него не затирает сохранённый чек-лист — сервер оставляет прежнее значение.

Ошибки всегда в формате `{ "error": "текст" }` с честным HTTP-статусом
(400 валидация, 401 не авторизован, 409 email занят, 429 слишком часто).

## Как устроена авторизация (шаг за шагом)

**Регистрация.** Пароль прогоняется через `scrypt` со случайной солью —
в базе лежит только хеш (`lib/password.ts`). Даже утечка базы не раскрывает
пароли: scrypt — медленная функция, перебор дорог, а соль обесценивает
готовые таблицы хешей.

**Вход.** Находим пользователя, сверяем хеш через `timingSafeEqual`
(сравнение за постоянное время). Если email не найден — всё равно считаем
хеш-«пустышку», чтобы по времени ответа нельзя было понять, существует ли
аккаунт. Сообщение об ошибке одно на оба случая: «Неверный email или пароль».

**Сессия.** Сервер подписывает JWT (`lib/jwt.ts`) и кладёт его в cookie:

- `httpOnly` — cookie не видна из JavaScript, XSS не украдёт токен;
- `sameSite=lax` — браузер не отправит её с POST-запросом с чужого сайта (CSRF);
- `secure` (в production) — только по HTTPS.

Сервер при этом **не хранит сессии**: вся информация — в самом токене,
подпись гарантирует подлинность. Цена такого подхода — токен нельзя
отозвать до истечения срока (см. «Идеи для развития»).

**Проверка.** `requireAuth` достаёт токен из cookie, проверяет подпись
и кладёт `req.userId`. Роуты прогресса работают только с данными этого
пользователя — «чужой» прогресс недостижим по построению.

## Решения, на которые стоит обратить внимание

- **Валидация на границе** (`schemas.ts`): всё из сети — недоверенные данные.
  Клиентская валидация — удобство, серверная — защита.
- **Параметризованный SQL** (`repositories/*`): значения подставляются через
  `?`, поэтому SQL-инъекция невозможна — ввод никогда не становится кодом.
- **Слои**: роуты не знают SQL, репозитории не знают HTTP. Поменять SQLite
  на Postgres = переписать два файла-репозитория.
- **Единый error handler**: роуты просто бросают `HttpError`/`ZodError`,
  формат ответа и логирование — в одном месте. Стектрейсы клиенту не отдаются.
- **`app.ts` отделён от `index.ts`**: приложение можно создать в тесте
  (supertest) без открытия порта.
- **PUT вместо PATCH для прогресса**: клиент шлёт полный снимок, запрос
  идемпотентен — ретраи безопасны.
- **Rate-limit на `/api/auth`**: пароли нельзя перебирать со скоростью сети.
- **Один origin**: в dev vite проксирует `/api`, в production статику раздаёт
  сам сервер. Поэтому нет ни CORS, ни проблем с cookie.

## Идеи для развития (в порядке усложнения)

1. **Refresh-токены**: короткоживущий access-токен + длинный refresh с
   хранением в базе — так сессию можно отозвать (сейчас logout лишь стирает cookie).
2. **Смена/сброс пароля** и подтверждение email.
3. **Версионированные миграции** вместо `CREATE TABLE IF NOT EXISTS`
   (первый шаг уже виден в `db.ts`: ALTER TABLE для колонки `tasks`).
4. **Интеграционные тесты** роутов через supertest + node:test.
5. **Переезд на Postgres** — проверка, что слои действительно разделены.
